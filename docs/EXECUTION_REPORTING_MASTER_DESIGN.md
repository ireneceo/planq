# PlanQ 실행·보고 통합 마스터 설계
### Project Execution Dashboard & Accountable Reporting System

> 30년차 서비스기획 · 경영컨설팅 · UI/UX 관점의 대기업 수준 통합 설계.
> 대상: 프로젝트 운영 캔버스(대시보드) 재설계 + 책임 기반 보고서(컨펌·롤업) 시스템.
> 기존 D1(조직)·D2(외부파트너)·D3(캔버스 v1)·#64(보고서 v1) 위에 재정렬.

---

## 0. Executive Summary

현재 캔버스/보고서는 기능이 빠르게 붙으며 **개념 혼선·정렬 불일치·중복**이 누적됐다. 본 설계는 셋을 바로잡는다.

1. **개념 정정** — "실행·추적"은 **업무(Task)**다. 계약·청구·세금계산서는 **거래(Billing)**이며 거래 탭으로 분리한다. 캔버스는 *전략 → 실행(업무 일정) → 산출물*의 한 흐름으로 정돈한다.
2. **전체 일정 진행 타임라인(★)** — 가로 축에 업무·마일스톤을 배치하고 "지금 여기"를 표시하는 단일 시그니처 그래프. 캔버스와 보고서가 **같은 컴포넌트**를 공유한다.
3. **책임 기반 보고 라인** — 멤버/프로젝트/부서 단위 보고서를 **자동 초안 → 책임자 검토·수정 → 확정**하고, 확정본이 **통합보고서로 자동 롤업**되어 대표가 본다. 주간 + 월간.

설계 원칙: **자동화 우선(빈 화면 0) · 사람이 책임지는 확정 · 단일 진실 원천 · 일관된 그리드 · 절제된 고급 비주얼.**

## 1. 설계 원칙 (Design Principles)

| # | 원칙 | 적용 |
|---|------|------|
| P1 | **개념 1:1** — 한 섹션 = 한 개념 | 실행추적=업무 / 거래=결제 분리. 워크스트림(묶음) ≠ 마일스톤(시점) ≠ 거래단계 |
| P2 | **자동화 우선** | 모든 보고/지표는 자동 초안이 항상 채워짐. 사람은 *검토·보정·확정*만 |
| P3 | **책임선 = 확정선** | 단위 보고서는 책임자가 확정해야 "최종". 통합은 확정본의 합 |
| P4 | **단일 진실 원천** | 진행률·health·델타는 `fetchProjectStats` 한 곳. 그래프·보고서·롤업이 동일 수치 |
| P5 | **일관 그리드** | 12-col 기반, 1/2/3열 정렬·간격(gutter 16) 통일. ad-hoc styled 금지 |
| P6 | **고급 비주얼 절제** | 디자인 토큰 100%, 그림자/모션 최소·일관. 정보 위계가 디자인을 이끈다 |

## 2. 정보 구조 (IA)

```
프로젝트
 └ 운영 캔버스(대시보드)  ← 프로젝트 1개의 전략·실행 현황 (편집)
     ├ 일정 진행 타임라인 ★
     ├ 전략 프레임
     ├ 핵심 추진과제(워크스트림)
     ├ 금주/차주 포커스
     ├ 산출물 · 이해관계자 · 리스크
     └ 관련 프로젝트
 └ 거래 탭        ← 계약·청구·세금계산서 (기존, 캔버스에서 분리 이전)
 └ 업무 탭        ← 업무 리스트/간트

보고 (워크스페이스)
 └ 단위 보고서 (자동초안 → 확정)
     ├ 멤버 보고   (책임=본인)
     ├ 프로젝트 보고 (책임=PM/owner)
     └ 부서 보고   (책임=부서장 lead_user_id)
 └ 통합보고서   ← 확정본 자동 롤업 (대표가 봄, 통합확정 설정 옵션)
     └ 주간 · 월간
```

핵심: **캔버스 = 살아있는 작업면 / 보고서 = 특정 시점 확정 스냅샷.** 보고서는 캔버스 데이터를 초안으로 끌어와 확정한다.

### 2.1 뷰별 목적 (Purpose Matrix) — "어디서 뭔지 딱 알게"
모든 화면은 **목적 한 문장**이 분명해야 하며, 그 목적에 봉사하는 콘텐츠만 둔다. 헷갈리는·중복·장식 콘텐츠는 배제.

| 뷰 | 목적 (한 문장) | 핵심 콘텐츠 (이것만) |
|----|---------------|---------------------|
| **프로젝트 대시보드/보고** | "이 프로젝트가 잘 가고 있나 — 이슈와 성과" | 일정 진행(타임라인)·진척률·주요 업무/마일스톤·리스크/이슈·성과(완료·산출물)·관련 프로젝트 |
| **멤버 보고** | "이 사람이 무엇을 했고 결과는" | 본인 업무 진행·완료·지연·기여·다음 주 계획 |
| **부서 보고** | "이 부서의 진행과 성과" | 부서 업무 롤업·멤버별 기여·부서 이슈·성과 |
| **통합보고서(대표)** | "회사 전체가 어떻게 가고 있나" | 확정본 롤업·포트폴리오·부서/멤버 요약·미확정 현황 |

원칙: **목적 외 콘텐츠는 넣지 않는다.** (예: 프로젝트 보고에 회사 전체 KPI 섞지 않음 / 멤버 보고는 "그 사람"에만 집중.)

## 3. 프로젝트 운영 캔버스 재설계

### 3.1 개념 정정 (가장 중요)
- **삭제:** 캔버스의 "실행·추적 / 로드맵·마일스톤"이 보여주던 *계약→청구→세금계산서* → **거래(billing stages)** 이므로 캔버스에서 제거. 거래는 기존 **거래 탭**이 담당.
- **삭제:** "업무 연계도(task_links)" → 오개념. **관련 프로젝트(project_links)** 로 대체.
- **신설:** 실행·추적의 본질 = **업무 일정 타임라인**(§3.3).

### 3.2 레이아웃 그리드 (정렬 표준)
- 컨테이너 `max-width: 1180px`, 중앙 정렬, 좌우 패딩 24.
- **12-col 그리드**, gutter 16. 섹션은 정해진 열 폭만 사용:
  - 1열(full): 타임라인, 워크스트림 보드
  - 2열(6/6): 전략(프레이밍·전략), 금주/차주
  - 3열(4/4/4): 산출물 · 이해관계자 · 리스크
- 카드 표준: `radius 12 · padding 16~18 · border #E2E8F0 · shadow 0 1px 2px rgba(15,23,42,.04)`. 모든 카드 **동일 높이 행 정렬**(grid `align-items: stretch`).
- 섹션 헤더 표준: `SectionTitle 14/700` + 우측 액션 슬롯. 섹션 간 수직 리듬 24.
- 반응형: ≤1024 → 2열을 1열, 3열을 2열; ≤640 → 전부 1열.

### 3.3 전체 일정 진행 타임라인 ★ (시그니처 컴포넌트 `ScheduleTimeline`)

**목적:** "지금 얼마나 왔고, 지금 주요 업무가 무엇인지"를 한눈에.

```
 시작┃──●───●───◆ 핵심──────┃▲지금┃──────○───◆ 핵심──○──┃마감
     │  └ 워크스트림 색 트랙(레인)                      │
 [전체 진행 37%]  [일정대비 ●순항/주의/지연]  [주요만 보기 ⃝]  [D-42]
```

**구성**
- **시간 축**: 프로젝트 `start_date~end_date`. 월/주 눈금. 가로 스크롤(좁을 때).
- **현재 마커("지금")**: 오늘 위치 수직선 + 라벨. 진행 위치 직관.
- **업무 표현**:
  - 기간 있는 업무 = 막대(start→due), 마감만 있으면 포인트.
  - **워크스트림 색**으로 묶음(레인 or 색). 워크스트림 = 묶음 축.
  - **마일스톤(주요 업무, `is_milestone`)** = 큰 다이아몬드 + 라벨(축 위).
- **진행 요약 바**(상단):
  - 전체 진행률 % (Σ진행률/업무수, `fetchProjectStats` 동일식)
  - **일정 대비 상태**: 기대 진행(경과기간 비율) vs 실제 진행 → 순항/주의/지연(green/yellow/red)
  - 남은 기간 D-, 지금 주요 업무 수
- **"주요만 보기" 토글**: 마일스톤(주요 업무)만 남김. 프로젝트 기본값은 `projects.timeline_key_only` 설정.
- **상호작용**: 포인트 클릭 → 업무 상세. 마일스톤 hover → 요약 툴팁.
- **공유**: 이 컴포넌트를 캔버스·프로젝트 보고서·통합 롤업이 모두 재사용(읽기 모드 prop).

**마일스톤 = "주요 업무" 모델 (워크스트림과 구분)**
- 워크스트림(핵심 추진과제) = 업무를 묶는 **상위 그룹**(색/레인).
- 마일스톤 = **주요 시점**으로 표시한 개별 업무(`tasks.is_milestone`).
- 관계: 업무는 워크스트림에 속하고(색), 그중 중요한 건 마일스톤으로 강조. 두 축이 직교 → 타임라인이 "묶음(색) × 시점(다이아몬드)"으로 읽힘.

### 3.4 캔버스 섹션 구성 (위→아래)
| 순서 | 섹션 | 열 | 내용 |
|---|------|----|------|
| 1 | 헤더 | full | 프로젝트명·status·기간·전체 진행률·owner/부서 |
| 2 | **일정 진행 타임라인 ★** | full | §3.3 |
| 3 | 전략 — 프레이밍 | 2열 | 추진배경·핵심과제 / 목표·성공지표 |
| 4 | 전략 — 가설 | full | 핵심 메시지(배너) + 추진 방식 |
| 5 | 핵심 추진과제(워크스트림) | full | 묶음 카드 + 진행바(타임라인 색과 동기) |
| 6 | 금주 / 차주 포커스 | 2열 | planned_week_start 기반 |
| 7 | 산출물 · 이해관계자 · 리스크 | 3열 | 정렬 통일 |
| 8 | **관련 프로젝트** | full | §3.5 |

거래(계약·청구·세금) = 캔버스에서 **제외**(거래 탭).

### 3.5b 워크스트림 = 업무 그룹 (양방향 동기화) ★
**핵심 추진과제(워크스트림)는 곧 "업무 그룹"이다.** 한 곳에서 만들면 모든 곳에 자동 반영·편집 가능.
- 캔버스에서 핵심 추진과제 추가 → **업무 메뉴(Task 리스트)에 그룹 헤더로 등장**, 업무를 그 그룹에 귀속(`task.workstream_id`).
- 업무 메뉴에서도 그룹(워크스트림) 추가/이름변경/순서변경/업무 드래그 이동 가능 → **캔버스에 즉시 반영**.
- 프로젝트 단위 업무는 **업무 그룹(워크스트림) 단위로 정렬·표시**가 기본(그룹 접기/펼치기, 그룹별 진행바).
- **단일 진실 원천**: `project_workstreams` 1테이블을 캔버스·업무리스트·타임라인·보고서가 공유.
- **실시간 동기화**(§16): 한 화면 변경 → `workstream:*`/`task:updated` broadcast → 모든 화면 즉시 갱신. 어디서든 추가/수정이 "당연히" 되도록.
- 미분류 업무는 "(그룹 없음)" 으로 묶어 노출. 그룹 삭제 시 소속 업무는 미분류로(데이터 보존).

### 3.5 관련 프로젝트 (Related Projects)
- **개념**: 프로젝트 ↔ 프로젝트 연결. 연결한 프로젝트만 카드로 표시(진행률·health·기간).
- **DB**: `project_links`(project_a_id < project_b_id, relation_label?, created_by) 양방향 1행.
- **UI**: 카드 그리드(연결된 프로젝트 status 요약 + 클릭 시 이동). "프로젝트 연결" 버튼 → 검색 선택.
- task_links는 캔버스에서 제거(업무 상세의 "관련 업무"로만 잔존).

## 4. 책임 기반 보고서 시스템

### 4.1 보고 라인
```
멤버 보고  (책임=본인)        ┐
프로젝트 보고(책임=PM/owner)  ├─ 확정 → 자동 롤업 → 통합보고서(대표)
부서 보고  (책임=부서장)      ┘                     ↳ 통합확정(설정 ON 시)
```
- 멤버=`WeeklyReview` 기 존재(확정 개념 보유) — 흡수.
- 프로젝트=책임자 `project.owner_user_id`(PM). 부서=`department.lead_user_id`.

### 4.2 보고서 생애주기 (단위 공통)
```
 ┌─ 자동 초안(live 집계, 항상 존재) ─┐
 │   cron이 주기 경계에 생성/갱신      │
 ▼                                   ▼
 [검토] → [수정: 서술·수치 보정] → [확정] → 스냅샷 고정(+confirmed_by/at)
                                      │
                                  [되돌리기] → 다시 draft (감사 기록 유지)
```
- **자동화 우선**: 미확정이어도 자동 초안으로 항상 채워짐(빈 화면 0).
- **수정 반영**: 책임자가 손대면 `edited_overrides`에 저장, 확정 시 *초안+수정* 합본이 스냅샷.
- **확정 = 책임**: confirmed_by(사람)·confirmed_at 기록. 이후 수치 변동과 무관하게 그 시점 박제.

### 4.3 주기 (주간 + 월간)
- 주간: `period_type='weekly'`, period_start=해당 주 월요일.
- 월간: `period_type='monthly'`, period_start=해당 월 1일.
- cron(`weeklyReviewCron` 확장): 주 경계 + 월 경계에 단위 초안 생성, 설정 시 미확정 자동확정.

### 4.4 통합보고서 (롤업)
- **자동 구성**: 같은 주기·기간의 확정된 멤버/프로젝트/부서 보고서를 취합.
- **확정/미확정 현황**: 단위별 확정 상태 표시. 미확정은 자동 초안으로 채우되 "미확정" 뱃지.
- **통합확정 설정**(`businesses.report_integrated_confirm` ON/OFF):
  - ON: 대표가 마지막 "통합 확정" 1회 → 통합본 박제(최종 보고자 존재).
  - OFF: 하위 확정 시 자동 노출(별도 통합확정 없음).
- 대표 화면: 읽기 중심 + (ON 시) 확정 버튼 + executive summary 편집.

### 4.5 미확정 처리
- 통합·상위 뷰에서 **자동 초안으로 채움** + "미확정/N건 대기" 표시. 비우지 않음.
- 마감 후에도 미확정이면 설정에 따라 자동확정(finalized_by='auto').

## 5. 데이터 모델 (DB)

### 5.1 신규 컬럼
| 테이블 | 컬럼 | 용도 |
|--------|------|------|
| `tasks` | `is_milestone BOOLEAN default false` | 주요 업무(마일스톤) 표시 |
| `projects` | `timeline_key_only BOOLEAN default false` | 타임라인 "주요만 보기" 기본값 |
| `businesses` | `report_integrated_confirm BOOLEAN default false` | 통합확정 단계 사용 여부 |
| `businesses` | `monthly_finalize_enabled BOOLEAN default true` | 월간 자동확정 ON/OFF |

### 5.2 신규 테이블 `project_links` (관련 프로젝트)
```
id, business_id, project_a_id, project_b_id (a<b 강제), relation_label STRING(40) null,
created_by, timestamps · UNIQUE(project_a_id, project_b_id)
```

### 5.3 신규 테이블 `report_units` (단위 보고서 — 컨펌·롤업 단일 원천)
```
id, business_id,
scope ENUM('member','project','department'),
scope_ref_id INT,                  // user_id | project_id | department_id
period_type ENUM('weekly','monthly'),
period_start DATEONLY,             // 월=1일, 주=월요일
status ENUM('draft','confirmed') default 'draft',
auto_snapshot JSON,               // 자동 초안 (live 집계 결과)
edited_overrides JSON null,       // 책임자 수정분 (병합 우선)
narrative TEXT null,              // 책임자 서술
confirmed_by INT null, confirmed_at DATE null, finalized_by ENUM('manual','auto') null,
timestamps · UNIQUE(business_id, scope, scope_ref_id, period_type, period_start)
```
- 멤버 보고는 기존 `WeeklyReview`와 정합: v2에서 report_units 로 단일화(마이그레이션), v1은 멤버=WeeklyReview 유지하고 report_units 는 project/department + 월간부터.
- 통합본은 기존 `BusinessWeeklyReport` 확장(주간) + 월간 row 추가, snapshot 에 단위 확정 상태 포함.

### 5.3 마이그레이션
- 컬럼 추가·신규 테이블 = sync-database 자동(ENUM은 신규 테이블에만). 백필 불필요.

## 6. API 설계

### 6.1 캔버스/타임라인
- `GET /api/projects/:id/timeline?key_only=` — 업무 일정 데이터(업무·마일스톤·워크스트림 색·기간·진행률·일정대비). ScheduleTimeline 전용 경량 집계.
- `PATCH /api/tasks/.../:id` 에 `is_milestone` 수용(담당/작성/owner/admin).
- `PATCH /api/projects/:id/settings` 에 `timeline_key_only`.
- `GET/POST/DELETE /api/projects/:id/links` — 관련 프로젝트 연결(같은 워크스페이스만, 자기 제외, 중복 차단). 멤버 전용.

### 6.2 보고서
- `GET /api/reports/:biz/unit?scope=&ref_id=&period_type=&period_start=` — 단위 보고서(없으면 자동 초안 생성·반환).
- `PATCH /api/reports/:biz/unit/:id` — 수정(narrative·overrides). 책임자만.
- `POST /api/reports/:biz/unit/:id/confirm` · `/reopen` — 확정/되돌리기. 책임자만. 감사·broadcast.
- `GET /api/reports/:biz/integrated?period_type=&period_start=` — 통합 롤업(확정/미확정 현황 + 취합).
- `POST /api/reports/:biz/integrated/confirm` — 통합확정(설정 ON·대표만).
- `PUT /api/businesses/:id/report-settings` — 통합확정·월간 토글.
- 권한: 단위 책임자 검증(멤버=본인 / 프로젝트=owner / 부서=lead). 통합=owner/admin. 멀티테넌트 WHERE business_id.

### 6.3 실시간/cron
- 단위 확정·수정 → `io.to('business:').emit('report:updated')` (§16). 통합 뷰 silentLoad.
- cron: 주/월 경계 단위 초안 생성 + 미확정 자동확정(설정).

## 7. UI/UX 명세

### 7.0 "리포트처럼" — 시각 품질 기준 (필수)
모든 대시보드·보고서는 **한 장의 잘 디자인된 리포트**다. 단순 나열 금지.
- **구조**: 헤더(제목·기간·상태) → 요약 시각화(KPI·그래프) → 디테일 순. 정보 위계가 크기·위치로 드러남.
- **시각 요소 필수**: 일정 진행 타임라인 · KPI 카드(숫자+델타+health 색) · 진행바 · 추세 스파크라인/도넛. 리스트도 카드/배지/색으로 시각화.
- **"나열만" 금지**: 텍스트 목록은 위계·색·아이콘·요약 수치로 재구성. 한눈에 읽히는 밀도.
- **인쇄/공유 품질**: 보고서는 PDF·공유로 그대로 내보내도 손색없는 완성도(여백·정렬·타이포 일관).
- 적용 대상: 프로젝트 대시보드, 단위 보고서(멤버·프로젝트·부서), 통합보고서, 그 외 모든 대시보드.

### 7.1 비주얼 시스템 (대기업 톤)
- 색: COLOR_GUIDE 토큰. Primary teal(#14B8A6 계열), 상태 green/amber/red(순항/주의/지연), 워크스트림 카테고리 팔레트(8색, 차트 전용).
- 타이포: title 18/700 · section 14/700 · body 13/500 · meta 11/600. 숫자(KPI) 22/700 tabular-nums.
- 카드/그림자/간격: §3.2 표준. 모션 0.15s ease, 과장 금지.
- 빈/로딩/에러: skeleton · 아이콘+1줄+CTA · 원인+해결. (alert/toast.success 금지)

### 7.2 ScheduleTimeline 컴포넌트 명세
- props: `tasks[], milestones[], workstreams[], start, end, today, keyOnly, readonly, onTaskClick`.
- 레이어: 축(눈금) → 워크스트림 색 트랙 → 업무 막대/포인트 → 마일스톤 다이아몬드 → 현재 마커 → 진행 오버레이.
- 접근성: 포인트 button+aria-label, 키보드 포커스, 색+형태 이중부호(색맹 대비).
- 반응형: 좁으면 가로 스크롤 + 좌측 "오늘로" 점프. 모바일 터치 타겟 ≥40.
- 재사용: 캔버스(편집 진입 가능) / 보고서(readonly).

### 7.3 보고서 UI
- 단위 보고 화면: 상단 상태칩(초안/확정·책임자·시각) + 자동 초안 본문(타임라인+KPI+하이라이트) + **수정 영역**(서술·보정, AutoSave) + **[확정] 버튼**(책임자만, 중복가드).
- 통합보고서: 단위 확정 현황 매트릭스(멤버/프로젝트/부서 × 확정/미확정) + 롤업 요약 + (설정 ON) [통합 확정].
- 주간/월간 토글, 기간 네비.

### 7.4 정렬·정돈 (현 문제 직접 해결)
- 캔버스 모든 섹션 §3.2 그리드로 재배치 — 1/2/3열 폭·gutter·카드 높이 통일.
- 섹션 제목·간격·아이콘 일관. ad-hoc inline style 제거.

## 10. 핵심 결정 박제
1. 실행추적=업무 / 거래=결제(거래 탭). 캔버스에서 거래 제거.
2. 업무연계도(task_links) 제거 → 관련 프로젝트(project_links).
3. 타임라인 ★ = 시그니처 공유 컴포넌트(캔버스+보고서). 현재 마커·마일스톤·주요만 토글.
4. 워크스트림(묶음) ≠ 마일스톤(주요업무 시점). 직교 2축.
5. 보고: 자동초안 → 책임자 수정 → 확정 → 통합 자동 롤업. 통합확정 설정 토글. 주간+월간. 미확정=자동초안+뱃지.
6. 책임자: 멤버=본인 / 프로젝트=owner / 부서=lead_user_id / 통합=대표(owner·admin).
7. **워크스트림 = 업무 그룹.** 단일 테이블 공유 + 양방향 동기화(캔버스↔업무리스트↔타임라인↔보고서). 어디서든 추가/수정 즉시 반영.
8. **뷰별 목적 1문장** 명확화 — 목적 외 콘텐츠 배제(프로젝트=이슈·성과 / 멤버=본인 진행·결과 / 부서=부서 진행·성과 / 통합=회사 전체).
9. **리포트급 비주얼** — 모든 대시보드·보고서는 디자인된 리포트(헤더→요약 시각화→디테일). 단순 나열 금지.
