## 1. 목적 · 사용자 시나리오

### 1.1 정정된 의사결정 (이번 세션 합의)
| 영역 | 결정 |
|---|---|
| **결제 채널** | 송금(계좌이체) 기본. PG 통합 X. PlanQ 는 청구 발행·안내만. |
| **세금계산서** | 사용자 외부 채널(홈택스·팝빌·회계사) 발행 → "발행 완료" 마킹. 자동 발행 X. |
| **분할 결제** | 발행자가 단일/분할 토글 선택. 분할 시 비율·일정 자유 입력. |
| **전자서명** | Phase 1 자체 구현 (이메일 OTP + 캔버스 + 명시 동의). 모두싸인 통합 X (Phase 2 옵션). |
| **이름** | 액션 = "**서명 받기**", 화면 = "**문서 서명**". 전자계약/공인 단어 회피. |

### 1.2 핵심 페르소나
| 역할 | 워크스페이스 | 행동 |
|---|---|---|
| **Owner / Member** | PlanQ 사용자 (컨설팅사·에이전시·프리랜서) | 견적·계약 작성, 서명 받기, 청구서 분할 발행, 채팅 결제 요청, 입금 확인 마킹, 세금계산서 마킹 |
| **Client (고객사)** | 외부 (PlanQ 미가입 가능) | 토큰 URL 또는 채팅 카드로 진입, 본인 OTP 인증 후 서명/거절, 송금 후 완료 알림 |

### 1.3 End-to-end 시나리오 (한 프로젝트)

```
[1] 영업
   사용자: Q docs > AI/템플릿 → 견적서 작성 → 공유
   고객: 토큰 URL → 검토 → 사용자가 "서명 받기" 보냄 → 고객 OTP·캔버스 서명
   ✓ Quote.status='accepted' → 사용자 화면에 "이 견적으로 청구서 발행" 후속 카드

[2] 계약
   사용자: Q docs > 계약서·SOW 작성 (견적 prefill)
   양사 "서명 받기" → 양사 모두 서명 완료
   ✓ Document.status='signed' → "분할 청구 일정 만들기" 후속 카드

[3] 청구
   사용자: Q Bill > 청구서 발행 (계약 prefill)
   분할 토글 ON → 선금 30 / 중도 40 / 잔금 30 (자유 비율)
   "선금" 행 → "결제 요청 보내기" → 채팅 카드 발송

[4] 입금
   고객: 채팅 카드 → /pay/:token → 계좌 정보 + 입금자명 가이드 → 송금 → "송금 완료 알림"
   사용자: 알림 → 입금 확인 → "결제 완료" 마킹
   ✓ installment.status='paid' + 카드 자동 갱신

[5] 세금계산서
   사용자 알림 ("세금계산서 발행해주세요")
   사용자: 외부 채널(홈택스·팝빌)에서 발행 → "발행 완료" 마킹 + 발행번호 입력

[6] 반복
   SOW 마일스톤 검수 통과 → 다음 installment 자동 활성 → 채팅 카드 → 위 루프
```

각 단계가 **자동 다음 행동을 제안** — 사용자 인지 부담 최소화.

### 3.2 활용 (수정 없음)

- `Document.signature_image_b64`, `signed_at`, `signed_ip`, `signature_data` (기존) — Document 모델용 서명은 그대로. Post 모델용은 신규 signature_requests 테이블 사용. **추후 통합 시 Document.signature_image_b64 도 signature_requests 로 마이그레이션**.
- `Message.kind='card', meta JSON` — card_type 에 `'signature_request'`, `'payment'` 추가.
- `BillEvent` (Q_BILL_SPEC) — invoice 결제·취소 audit. 그대로 사용 + installment 이벤트 추가.

### 3.3 의도된 설계 결정

**Q1. 서명 요청 테이블이 entity_type 으로 Post / Document 양쪽을 가리키는 이유**

→ Post(에디터 자유 양식), Document(모델 정형 견적·NDA 등) 둘 다 서명이 필요. 하나로 통합해서 신규 entity 추가 시 (예: Quote 별도 PDF) 도 같은 테이블 재사용.

**Q2. invoice_installments 분리 이유 (Invoice 안에 JSON 으로 둘 수도 있는데)**

→ status 별 인덱싱 (overdue cron, 결제 마킹), JOIN 으로 알림 생성, 회계 라인 단위 추적, 향후 마일스톤 ref 등 확장성. JSON 보다 관계 테이블이 정합성·검색에 유리.

**Q3. payment_requests 가 invoice_installments 와 별도인 이유**

→ "quick 결제 요청" (invoice 미수반·작은 금액) 케이스 지원. 또한 한 installment 를 여러 번 채팅에 보낼 수 있음 (1차 알림 후 catch-up). payment_requests 가 channel 인스턴스, installment 가 회계 단위.

## 4. API 설계

### 4.1 서명 받기 (Phase A)

| Method | Path | 권한 | 설명 |
|---|---|---|---|
| POST | `/api/posts/:id/signatures` | 멤버 | 서명 요청 N명 발급 (이메일+카드 발송), `signature_requests` rows 생성 |
| GET | `/api/posts/:id/signatures` | 멤버 | 그 문서의 서명 진행 목록 |
| DELETE | `/api/signatures/:id` | requester or owner | 서명 요청 취소 (status='canceled') |
| POST | `/api/signatures/:id/reminder` | requester | 만료 전 재발송 |
| GET | `/api/sign/:token` | **공개** | 토큰 검증 + 문서 본문 + 진행 상태 |
| POST | `/api/sign/:token/otp` | **공개** | OTP 발송 (6자리, 5분 유효, 5회 실패 시 lock) |
| POST | `/api/sign/:token/verify` | **공개** | OTP 검증 → otp_verified_at 기록 |
| POST | `/api/sign/:token/sign` | **공개** (verify 필수) | 서명 이미지 + 동의 → signed_at 저장. 양사 모두 signed → entity status='signed' |
| POST | `/api/sign/:token/reject` | **공개** (verify 필수) | 거절 + 사유 |

**보안**:
- token: `crypto.randomBytes(32).toString('hex')` 64자
- OTP: 6자리 숫자, sha256 해시 저장, TTL 5분, 시도 5회 lock
- IP·UA 모든 단계 audit
- HTTPS 만 허용 (이미 nginx 설정됨)
- 동일 IP 에서 다중 OTP 요청 rate limit (1분 3회)

### 4.2 Q Bill 분할 청구 (Phase B)

| Method | Path | 권한 | 설명 |
|---|---|---|---|
| GET | `/api/invoices` | 멤버 | 리스트 (status·date 필터·검색) |
| POST | `/api/invoices` | 멤버 | 발행 — body 에 `installment_mode`, `installments[]` 포함 시 분할 자동 생성 |
| GET | `/api/invoices/:id` | 멤버 | 상세 (installments include) |
| PUT | `/api/invoices/:id` | requester or owner | 수정 (draft 만 허용) |
| DELETE | `/api/invoices/:id` | requester or owner | 삭제 (draft 만) |
| POST | `/api/invoices/:id/send` | 멤버 | 발행 확정 → status='sent', share_token 발급 |
| GET | `/api/invoices/:id/installments` | 멤버 | 분할 목록 |
| PATCH | `/api/invoices/:id/installments/:iid` | 멤버 | 결제 완료 / 취소 / 메모 갱신 |
| POST | `/api/invoices/:id/installments/:iid/mark-paid` | 멤버 | "결제 완료" 마킹 (paid_at·marked_by 기록) |
| POST | `/api/invoices/:id/installments/:iid/mark-tax-invoice` | 멤버 | 세금계산서 발행 마킹 (발행번호·발행일자 입력) |
| GET | `/api/public/invoices/:token` | **공개** | 청구서 + 분할 일정 표시 |

**Cron**:
- 매일 00:30 — installments.due_date 가 오늘 이전이고 status='sent' 면 'overdue' 로 전환 + 알림 생성

### 4.3 채팅 결제 요청 (Phase C)

| Method | Path | 권한 | 설명 |
|---|---|---|---|
| POST | `/api/conversations/:cid/payment-requests` | 멤버 | quick 발행 또는 invoice 연결 → 카드 메시지 발송 |
| GET | `/api/payment-requests/:id` | 멤버 | 상세 |
| DELETE | `/api/payment-requests/:id` | requester or owner | 취소 |
| GET | `/api/pay/:token` | **공개** | 결제 안내 페이지 데이터 |
| POST | `/api/pay/:token/notify` | **공개** | "송금 완료 알림" 클릭 → notified_at 기록 + 사용자 알림 생성 |
| POST | `/api/payment-requests/:id/mark-paid` | 멤버 | 사용자 결제 완료 마킹 → 카드 메시지 자동 갱신 (Socket.IO) |

### 4.4 통합 트리거 (Phase D)

내부 이벤트 시스템 (BillEvent + Notification):

| 트리거 | 이벤트 | 후속 액션 |
|---|---|---|
| signature_request.signed (양사 모두) | entity status='signed' | 사용자 알림 + UI 사이드바에 후속 액션 카드 ("청구서 발행" / "분할 일정 만들기") |
| signature_request.rejected | entity status='rejected' | 요청자 알림 + 사유 표시 |
| invoice_installment.marked_paid | BillEvent.paid_partial | 카드 메시지 갱신 + 알림 + 다음 installment cron 활성 검토 |
| project.milestone_status='completed' | Custom event | 그 milestone 에 ref 된 installment.status='sent' 자동 활성 (옵션) |
| installments.overdue (cron) | Notification(installment_overdue) | 사용자에게 연체 알림 |

### 4.5 알림 센터 API

| Method | Path | 권한 | 설명 |
|---|---|---|---|
| GET | `/api/notifications` | 본인 | 본인 알림 목록 (unread first, paginated) |
| PATCH | `/api/notifications/:id/read` | 본인 | 읽음 처리 |
| POST | `/api/notifications/mark-all-read` | 본인 | 전체 읽음 |

## 7. 엣지 케이스 매트릭스

### 7.1 서명

| 케이스 | 처리 |
|---|---|
| 토큰 잘못된/만료 | 404 + "공개되지 않았거나 만료된 링크입니다" |
| OTP 5회 실패 | 토큰 lock 60분, 재발송도 차단 |
| OTP 5분 만료 | "만료됨, 재발송" 버튼 |
| 서명 후 다시 진입 | "이미 서명 완료" 화면 + 거절은 변경 불가 |
| 한 명 거절 → 양사 진행? | 정책 결정: 거절 시 entity status='rejected' 즉시, 다른 서명자 차단 |
| 만료일 초과 | cron 으로 status='expired' + 재발송 옵션 |
| 요청자가 취소 | status='canceled' + 토큰 무효 |
| 같은 이메일에 중복 발송 | 이전 row status='canceled' + 새 row 생성 |
| 동의 박스 미체크 | "서명하기" 비활성 |
| 서명 캔버스 빈 그림 | "서명을 그려주세요" 검증 |

### 7.2 분할 청구

| 케이스 | 처리 |
|---|---|
| 분할 비율 합계 ≠ 100% | 발행 차단 + 메시지 |
| installment 금액 합계 ≠ invoice.total | 자동 재계산 + 경고 |
| 분할 발행 후 invoice 금액 변경 | draft 만 허용. sent 후 변경 불가 (취소 후 재발행) |
| installment 결제 완료 후 마킹 취소 | 가능 (audit 기록) |
| 모든 installment paid → invoice.status='paid' 자동 |
| 일부 paid → 'partially_paid' |
| 기한 초과 cron | 'sent'→'overdue', 알림 생성 |
| 세금계산서 발행번호 중복 입력 | 정합성 X (warn only) |

### 7.3 채팅 결제

| 케이스 | 처리 |
|---|---|
| 채팅방에 client_id 없음 (internal 채널) | 결제 요청 버튼 비활성 + tooltip |
| invoice 가 다른 client 의 것 | 발송 차단 |
| 동일 installment 중복 발송 | 허용 (catch-up). 이전 카드는 stale 표시 |
| 토큰 만료 (없음 — 만료 정책 추가 검토) | invoice 만료일과 동기 |
| 송금 완료 알림 다중 클릭 | 1회만 기록, 이후 disable |
| 사용자 마킹 후 입금 누락 발견 | 마킹 취소 가능 + audit |
| Socket.IO 실패 (재연결) | refresh 시 카드 status 자동 동기 |

### 7.4 공통

| 케이스 | 처리 |
|---|---|
| 워크스페이스 owner 가 아닌 멤버 액션 | 권한 분리 (자세한 분 §8.3) |
| 클라이언트 IP 변경 (모바일↔PC) | OTP 재인증 안 함 (token 유효하면 OK) |
| 본인 인증 후 브라우저 닫음 | otp_verified_at 유지, 재진입 시 단계 3 부터 |
| 동시성 (양사 동시 서명) | DB transaction + row lock (FOR UPDATE) |

## 8. 보안 · 법적 검토

### 8.1 한국 전자서명법 (2020 개정) 충족

| 요건 | 본 설계 충족 |
|---|---|
| ① 양 당사자 합의 | 서명 시 동의 박스 필수 체크 |
| ② 본인 확인 | 이메일 OTP (6자리, sha256 해시, 5분 TTL, 5회 lock) |
| ③ 위·변조 방지 | signature_image_b64 sha256 + signed_at·ip·ua audit 불변 |
| ④ 보존 (10년) | DB 영구 보관 + Document/Post 삭제 시에도 signature_requests 유지 |

→ 80% 비즈니스 계약 효력 충분. 큰 분쟁 우려 시 모두싸인 통합 (Phase 2 옵션).

### 8.2 개인정보

- 서명자 이메일·이름·서명 이미지 = 개인정보
- 처리 근거: 계약 이행 (정보주체 동의 X 가능)
- 보유 기간: 계약 종료 후 5년 (전자상거래법 제6조)
- 가공 (서명자 이름·이메일 audit log) 명시 동의 박스에 포함

### 8.3 권한 매트릭스

| 액션 | owner | admin | member | client |
|---|:---:|:---:|:---:|:---:|
| 서명 요청 발송 | ✓ | ✓ | ✓ | — |
| 서명 요청 취소 | ✓ | ✓ | 본인 발송분만 | — |
| 청구서 발행 | ✓ | ✓ | ✓ | — |
| 청구서 수정 (draft) | ✓ | ✓ | ✓ | — |
| 청구서 발송 (sent) | ✓ | ✓ | ✓ | — |
| 청구서 삭제 | ✓ | ✓ | 본인 발행분만 | — |
| installment 마킹 | ✓ | ✓ | ✓ | — |
| 결제 요청 발송 | ✓ | ✓ | ✓ | — |
| 결제 카드 보기 | — | — | — | ✓ (token) |
| 서명 페이지 접근 | — | — | — | ✓ (token+OTP) |

### 8.4 토큰 보안

- 길이: 32 bytes hex (64 chars) → 2^256 추측 불가
- 발급 후 expires_at 까지만 유효
- 1회 사용 (서명 완료/거절 시 lock)
- 결제 token 은 invoice.due_date + 7일 후 만료
- 모든 토큰 페이지: HTTPS only, no-cache, no-index meta

### 8.5 Rate Limit (신규 미들웨어 추가)

| 엔드포인트 | 제한 |
|---|---|
| POST /sign/:token/otp | IP 1분에 3회, token 5분에 5회 |
| POST /sign/:token/verify | token 5분에 5회 (실패 시 lock) |
| POST /pay/:token/notify | token 1회만 |

## 9. 성능 · 확장

### 9.1 인덱스
- signature_requests: token UNIQUE, (entity_type,entity_id), (status,expires_at)
- invoice_installments: (invoice_id,installment_no UNIQUE), (status,due_date), (milestone_ref)
- payment_requests: token UNIQUE, (conversation_id,status), (invoice_id,installment_id)
- notifications: (user_id,read_at,created_at), (created_at DESC)

### 9.2 Cron
- 매일 00:30 — overdue 전환 + 알림 생성 (1 query)
- 매일 09:00 — 만료 임박 (3일 전) 알림
- 매시간 — expired 토큰 정리

### 9.3 부하 예상 (Pro 플랜 1만 워크스페이스 × 월 10건)
- 청구서 100,000 row/월, installments 평균 3 = 300,000 row/월
- signature_requests 50,000 row/월
- DB 1년 ~3M row → 인덱스 충분히 처리

### 10.1 단위 테스트 (백엔드)

### 10.2 E2E (Node 스크립트)
- 서명: 발급 → OTP → 검증 → 서명 → 양사 status=signed
- 분할: 발행 → installments 3개 → 1개 마킹 → invoice partially_paid → 모두 마킹 → paid
- 결제 카드: quick 발행 → 카드 메시지 → 공개페이지 GET → notify → 마킹 → 카드 갱신
- 거절 / 만료 / lock / 권한 차단

### 10.3 UI 시나리오
- 견적 서명 → 청구서 발행 후속 카드 → 채팅으로 보내기 → 입금 → 마킹 → 세금계산서
- 알림 센터 표시·읽음·점프

## 13. 결정 필요 항목 (구현 전 확정)

1. **거절 정책**: 한 서명자 거절 시 즉시 entity status='rejected' (다른 서명자 차단) vs 다른 서명자도 진행 가능?
   → **추천**: 즉시 rejected (계약 의미상)

2. **결제 token 만료**: invoice.due_date + 7일 vs 항상 유효?
   → **추천**: due_date + 30일 (catch-up 가능)

3. **마일스톤 ↔ installment 자동 연결**: SOW 검수 통과 시 자동 활성?
   → **추천**: Phase D 옵션 토글 (워크스페이스 설정)

4. **알림 발송 채널**: in-app 만 vs 이메일 알림 (옵션)?
   → **추천**: in-app 디폴트 + 사용자 설정에서 이메일 옵션 (Phase D 후속)

5. **bank_account 필드**: Business 에 직접 추가 vs 신규 BankAccount 테이블?
   → **추천**: Business 에 단일 계좌 필드 추가 (단일 계좌 충분, 다중은 향후)
