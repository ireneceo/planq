# /기능설계 — 대규모 신규 기능 설계 스킬

대규모 신규 기능/시스템을 **6단계 구조화된 설계**로 진행하는 스킬.
각 단계 완료 시 사용자 승인을 받고 다음 단계로 진행. 최종 산출물은 `docs/` 폴더에 저장.

---

## 이 스킬을 언제 쓰나

**쓰는 경우 (대규모 신규 시스템):**
- 신규 DB 모델 3개 이상
- 신규 라우트/컨트롤러 파일 여러 개
- 역할별 UI 페이지 추가
- 외부 연동, 결제, 이메일 파이프라인 등

**쓰지 않는 경우 (작/중 규모):**
- 버그 수정, 텍스트 변경
- 단일 기능 추가/수정
- 이럴 땐 기존 워크플로우(요구사항 → UX → 기술 설계 → 구현 → 검증)로 충분

---

## 6단계 설계 흐름

### 1단계: 기능 정의
- 기능명 / 목적 / 핵심 사용자 / 핵심 유스케이스 / 기존 시스템과의 관계 / 성공 기준 / 비범위
- 완료 조건: 사용자가 "이 방향 맞다" 확인

### 2단계: API 구조 설계
- 엔드포인트 (METHOD /api/path)
- 역할 / 인증 / 요청 Body / 응답 / 에러 케이스
- 기존 라우트 파일 중 어디에 넣을지 지정
- 완료 조건: 엔드포인트 목록 + spec 확정, 사용자 승인

### 3단계: DB 구조 정의
- 테이블명 / 컬럼 / 인덱스 / ENUM / 관계
- 모델 파일 + association 추가 계획
- 마이그레이션 전략
- 완료 조건: 스키마 확정, 사용자 승인

### 4단계: UI 흐름 정의
- 페이지 목록 (List/Detail/Form)
- 사이드바 메뉴 / 라우트 / 역할 가드
- 페이지별 상태 + 데이터 소스
- 모달 / 빈 상태 / 로딩/에러 / 반응형 / i18n
- 기존 공통 컴포넌트 재사용
- 완료 조건: 페이지 목록 + 와이어프레임 확정, 사용자 승인

### 5단계: 코드 생성
- 모델 → 마이그레이션 → 라우트 → 서버 등록 → 인터페이스 → API 클라이언트 → 페이지 → 사이드바 → 라우트 → 가드 → i18n → 이메일
- 2~4단계 spec 엄격 준수 ("임의 추가 금지")
- 각 파일 완료 후 간단 보고
- 완료 조건: 빌드 성공 + 타입에러 0건

### 6단계: 테스트 시나리오
- 정상 흐름 (Write→Read→Update→Delete 왕복)
- 권한 테스트 (다른 역할/tenant 접근 → 403/404)
- 경계 케이스 (0, null, 빈, 최대, 중복)
- 에러 케이스 (필수 누락, 타입 오류, 존재하지 않는 FK)
- 연관 영향 확인
- 임시 Node 스크립트로 실제 HTTP 호출 → 검증 후 스크립트 삭제
- 완료 조건: 모든 시나리오 통과

---

## 설계 문서 저장

- `docs/<FEATURE_NAME>_DESIGN.md` — 1~4단계 산출물
- `docs/<FEATURE_NAME>_TESTS.md` — 6단계 산출물
- 기존 문서 있으면 추가, 새 파일 만들지 말기
- 1문서 = 1주제

---

## 흐름 요약

사용자가 `/기능설계 <기능명>` 호출
→ 1단계 기능 정의 표 → 승인
→ 2단계 API → 승인
→ 3단계 DB → 승인
→ 4단계 UI → 승인
→ docs/ 저장
→ 5단계 코드 구현 (단계별 보고)
→ 6단계 테스트 실행
→ 기존 /검증 스킬 실행
→ 완료

---

## 절대 금지
- 1~4단계 생략하고 바로 코딩
- 설계에 없는 기능 임의 추가
- 승인 없이 다음 단계 진행
- 6단계 테스트 건너뛰고 "완료" 보고
- 작은 작업에 이 스킬 사용 (오버헤드)
