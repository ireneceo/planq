# 검증 명령어

구현 완료 후 반드시 실행하는 검증 프로세스.
모든 단계를 실제로 실행해야 하며 확인했습니다만 말하는 것은 금지.
각 파트별 전문가 관점에서 체크하고 실제 결과를 보여줘야 한다.

## 0단계: 헬스체크 (필수, 가장 먼저 실행)

```bash
cd /opt/planq/dev-backend && node /opt/planq/scripts/health-check.js
```

- 종료 코드 0: 핵심 기능 모두 정상 → 검증 진행
- 종료 코드 0이 아님: **검증 실패. 깨진 항목 먼저 복구해야 함.** 깨진 채로 9단계 검증 진행 금지.
- 결과를 사용자에게 표시할 것 (✓/✗ 표).

## 검증 체계: 5개 파트 전문가 관점

개발기획: 요구사항 충족, 데이터 흐름 정합성, 엣지케이스 (3단계, 6단계)
백엔드 개발: API 정상, DB 정합성, 보안, 에러 처리 (2단계, 3단계)
프론트엔드 개발: 빌드, 타입 에러, 렌더링, 상태 관리 (1단계, 4단계)
UI/UX 기획 디자인: 디자인 통일성, 인터랙션, 빈 상태, 로딩 (8단계)
QA: 유저 흐름, 역할별, 반응형, 연관 영향 (5단계, 7단계, 9단계)

## 9단계 검증 (건너뛰기 절대 금지)

1단계 빌드: cd /opt/planq/dev-frontend && npm run build (run_in_background), 타입에러 0건 확인
2단계 백엔드: pm2 restart planq-dev-backend, 에러로그 확인, 인증/응답형식 체크
3단계 API 실호출: Node 스크립트로 실제 HTTP 호출 (포트 3003). Write->Read 왕복, 에러케이스, 권한별 접근
4단계 프론트 렌더링: curl로 HTML/JS 번들 확인 (https://dev.planq.kr)
5단계 유저 흐름: 역할별 로그인->기능->저장->재조회 순서대로 API 호출
6단계 요구사항 대조: 요청 항목별 구현 완료 여부 표시
7단계 연관 영향: 공통 컴포넌트 사용처 확인, 부작용 없음 확인
8단계 UI/UX 품질: **아래 8-A ~ 8-G 전부 수행** (트렌디/고퀄리티 디자인 + 기능 적합성 디테일)
9단계 크로스페이지: 네비게이션, SPA 라우팅, 역할별 리다이렉트

## 8단계 UI/UX 품질 — 상세 템플릿

참조 문서:
- `/opt/planq/dev-frontend/UI_DESIGN_GUIDE.md` (레이아웃·컴포넌트·alert 금지·자동저장 등 규칙 원본)
- `/opt/planq/dev-frontend/COLOR_GUIDE.md` (색상 토큰 원본)

### 8-A 디자인 시스템 준수

- **색상 (COLOR_GUIDE.md 기준)**
  - Primary (Deep Teal): `#14B8A6` (500) / `#0D9488` (600, hover) / `#0F766E` (700, press) / `#F0FDFA` (50, selected bg)
  - Accent (Rose/Coral CTA·AI 감지): `#F43F5E`
  - Text: `#0F172A` (title) / `#334155` (body) / `#64748B` (subtle) / `#94A3B8` (placeholder)
  - Border: `#E2E8F0` (default) / `#CBD5E1` (strong)
  - Background: `#FFFFFF` (card) / `#F8FAFC` (page)
  - Status: Success `#22C55E` / Warning `#F59E0B` / Error `#EF4444`
  - **금지**: COLOR_GUIDE.md 에 없는 hex 신규 도입, 인라인 hex 남발
- **폰트**
  - 크기 계층: title 18/700, body 14/500, label 13/500, caption 12/500, meta 11/500
  - weight 은 500/600/700 외 금지 (400 본문, 900 로고 외)
- **간격 (8의 배수)**
  - 8 / 12 / 16 / 20 / 24 / 28 px 만. 임의 px(7, 13, 22 등) 금지
  - Section padding 20-28px, Card padding 14-20px
- **Border radius**
  - Input 6-8px / Card 10-14px / Section 12-16px / Chip·Badge 16-20px (pill 시 999px)
  - 페이지 헤더·패널 헤더는 `PageShell`/`PanelHeader` 공통 컴포넌트 사용 — 직접 styled 선언 금지
- **Shadow**
  - 가벼운 box-shadow 만 사용 (`0 1px 2px rgba(0,0,0,0.05)` / `0 4px 12px rgba(0,0,0,0.06)`). 임의 커스텀 섀도우 금지

### 8-B 트렌드/고퀄리티 디테일

- **마이크로 인터랙션**: hover transition 0.15s, focus ring = Primary 500 + opacity 0.3 box-shadow, active 시 subtle scale/bg 변화
- **스위치/토글**: 라벨 + 보조 설명 조합, disabled 시 opacity 0.5, `role="switch"` + `aria-checked` 명시
- **빈 상태**: 단순 "No data" 문자열 금지 → Icons.tsx 아이콘 + 1줄 설명 + 명확한 CTA 버튼
- **로딩 상태**: 단순 "Loading..." 금지. spinner 또는 skeleton 사용, 레이아웃 변동(CLS) 없도록 placeholder 높이 고정
- **에러 상태**: 사용자에게 원인 + 해결 힌트. `Failed` 한 마디 금지
- **폼 저장**: AutoSaveField 적용 원칙 (UI_DESIGN_GUIDE.md 섹션 7) — 저장 버튼 없이 debounce 저장, ✓ 뱃지만 2초 페이드. 예외: 청구서 같은 복합 폼
- **긍정 피드백**: **alert, toast.success, window.alert 절대 금지** (헬스체크가 차단). 성공 시 모달 닫기 + 조용한 리프레시

### 8-C 기능 적합성

- **필드 타입 최적화**
  - 날짜 단일·범위: `components/Common/CalendarPicker.tsx` 사용. `<input type="date">` 직접 사용 지양 (허용 시 i18n/tz 일관성 확인)
  - 셀렉트: `PlanQSelect` 강제 (raw `<select>` / react-select 직접 import 금지 — 헬스체크 린트 차단)
  - 전화번호: `PhoneInput`
  - 타임존: `TimezoneSelector`
  - 파일 업로드: `FileUpload`
- **타임존 (PlanQ 필수 규칙)**
  - 모든 사용자 대면 시각은 `hooks/useTimeFormat` 또는 `utils/dateFormat.ts` 경유 — 직접 `toLocaleString()`/`toLocaleDateString()`/`toLocaleTimeString()` 호출 시 반드시 `timeZone` 옵션 포함
  - DB 는 UTC, 표시는 `user.workspace_timezone` 기준
  - 업무 경계(오늘/이번 주/연체) 계산도 워크스페이스 tz (`utils/timezones.ts` `todayInTz` 등 사용)
- **긴 폼**: 4개 이상 섹션이면 탭 분할 (`components/Common/TabComponents.tsx`) 또는 스텝
- **동적 리스트**: 추가/삭제 아이콘 버튼, 빈 배열 CTA, 카운트 배지 (Primary 100 bg + Primary 800 text)
- **체크리스트/설정**: 12개 이상은 카테고리 그룹핑 (Section + subtitle)
- **read-only**: `disabled` input 대신 definition list 스타일 (label + value), 값 없음은 "—" 플레이스홀더

### 8-D 반응형

- `@media (max-width: 768px)` 분기: 3컬럼 → 1컬럼, grid 1fr, padding 감소(24→16), 테이블 → 카드 변환
- 가로 스크롤 금지. 탭 오버플로우는 `overflow-x: auto` + scrollbar 숨김
- 모바일 입력 UX: `inputMode` 속성 (numeric/decimal/email 등), 터치 타겟 44px+ 확보
- 확인 방법: `curl https://dev.planq.kr/<path>` + 빌드 CSS 에 `@media` 분기 grep

### 8-E i18n (필수)

- **하드코딩 텍스트 0건** — 모든 문구 `t('ns:key')` 사용. 검사:
  ```bash
  grep -rEn "(['\"\`])[^'\"\`]*[가-힣][^'\"\`]*\1" dev-frontend/src --include='*.tsx' --include='*.ts' \
    | grep -v -E '//|/\*|\*/|^\s*\*' | grep -v '/locales/'
  ```
  결과 줄 없어야 통과
- ko/en 양쪽 JSON 에 키 존재 확인 (`public/locales/ko`, `public/locales/en`)
- `i18n.ts` 의 `ns` 배열에 신규 네임스페이스 등록 여부

### 8-F 접근성 기본

- `<button>` 에 `type="button"` (submit 아닌 경우)
- 아이콘 전용 버튼은 `title` + `aria-label`
- 토글: `role="switch"` + `aria-checked`
- 색상 대비 AA (텍스트 4.5:1)
- 포커스 ring 제거 금지 (Primary 500 opacity 0.3)

### 8-G 실제 확인 방법 (코드 리뷰만 금지)

- 변경한 페이지를 curl 로 직접 서빙 확인: `curl -s -o /dev/null -w "%{http_code}" https://dev.planq.kr/<path>`
- **구체적 파일:라인 인용**하며 항목별 ✅/❌ 기재 ("LayoutX.tsx:142 에 인라인 hex `#AABBCC` 남음 → COLOR_GUIDE.md 800 으로 교체 필요")
- 부적합 발견 시 즉시 수정 → 재빌드 → 재검증. "다음에 하겠다" 금지
- 빌드 산출물에서 인라인 hex 분포 확인:
  ```bash
  grep -rEn "#[0-9A-Fa-f]{3,8}" dev-frontend/src --include='*.tsx' --include='*.ts' \
    | grep -v -E '//|/\*' | wc -l
  ```
- i18n 하드코딩 검사 (8-E 스니펫)
- 레이아웃 표준 위반 검사 (페이지 루트에 `<Page>`/`<Header>` styled 직접 선언했는지):
  ```bash
  grep -rn "const Page = styled\|const Header = styled" dev-frontend/src/pages
  ```

## 절대 금지

- 코드 확인만으로 정상이라 하지 말고 실제 호출 결과를 보여라.
- API 호출 없이 검증 완료 금지. 빌드 안 돌리고 빌드 성공 금지.
- 에러 있는데 완료 금지. UI/UX 건너뛰기 금지.
- 8단계를 "색상 통일" 한 줄로 때우기 금지 — 8-A~8-G 전부 구체 결과 필요.
