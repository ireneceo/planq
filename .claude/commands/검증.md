# 검증 명령어

구현 완료 후 반드시 실행하는 검증 프로세스.
모든 단계를 실제로 실행해야 하며 확인했습니다만 말하는 것은 금지.
각 파트별 전문가 관점에서 체크하고 실제 결과를 보여줘야 한다.

## 0단계: 헬스체크 (필수, 가장 먼저 실행)

```bash
cd /opt/planq/dev-backend && node /opt/planq/scripts/health-check.js
```

- 종료 코드 0: 핵심 기능 모두 정상 → 검증 진행
- 종료 코드 0이 아님: **검증 실패. 깨진 항목 먼저 복구해야 함.** 깨진 채로 9단계 검증 진행 금지.
- 결과를 사용자에게 표시할 것 (✓/✗ 표).

## 검증 체계: 5개 파트 전문가 관점

개발기획: 요구사항 충족, 데이터 흐름 정합성, 엣지케이스 (3단계, 6단계)
백엔드 개발: API 정상, DB 정합성, 보안, 에러 처리 (2단계, 3단계)
프론트엔드 개발: 빌드, 타입 에러, 렌더링, 상태 관리 (1단계, 4단계)
UI/UX 기획 디자인: 디자인 통일성, 인터랙션, 빈 상태, 로딩 (8단계)
QA: 유저 흐름, 역할별, 반응형, 연관 영향 (5단계, 7단계, 9단계)

## 10단계 검증 (건너뛰기 절대 금지)

1단계 빌드: cd /opt/planq/dev-frontend && npm run build (run_in_background), 타입에러 0건 확인
2단계 백엔드: pm2 restart planq-dev-backend, 에러로그 확인, 인증/응답형식 체크
3단계 API 실호출: Node 스크립트로 실제 HTTP 호출 (포트 3003). Write->Read 왕복, 에러케이스, 권한별 접근
4단계 프론트 렌더링: curl로 HTML/JS 번들 확인 (https://dev.planq.kr)
5단계 유저 흐름: 역할별 로그인->기능->저장->재조회 순서대로 API 호출
6단계 요구사항 대조: 요청 항목별 구현 완료 여부 표시
7단계 연관 영향: 공통 컴포넌트 사용처 확인, 부작용 없음 확인
8단계 UI/UX 품질: **아래 8-A ~ 8-G 전부 수행** (트렌디/고퀄리티 디자인 + 기능 적합성 디테일)
9단계 크로스페이지: 네비게이션, SPA 라우팅, 역할별 리다이렉트
10단계 Playwright MCP e2e (옵션 — `/검증 --e2e` 또는 큰 사이클 마지막에): **아래 10단계 섹션 참조**

## 8단계 UI/UX 품질 — 상세 템플릿

참조 문서:
- `/opt/planq/dev-frontend/UI_DESIGN_GUIDE.md` (레이아웃·컴포넌트·alert 금지·자동저장 등 규칙 원본)
- `/opt/planq/dev-frontend/COLOR_GUIDE.md` (색상 토큰 원본)

### 8-A 디자인 시스템 준수

- **색상 (COLOR_GUIDE.md 기준)**
  - Primary (Deep Teal): `#14B8A6` (500) / `#0D9488` (600, hover) / `#0F766E` (700, press) / `#F0FDFA` (50, selected bg)
  - Accent (Rose/Coral CTA·AI 감지): `#F43F5E`
  - Text: `#0F172A` (title) / `#334155` (body) / `#64748B` (subtle) / `#94A3B8` (placeholder)
  - Border: `#E2E8F0` (default) / `#CBD5E1` (strong)
  - Background: `#FFFFFF` (card) / `#F8FAFC` (page)
  - Status: Success `#22C55E` / Warning `#F59E0B` / Error `#EF4444`
  - **금지**: COLOR_GUIDE.md 에 없는 hex 신규 도입, 인라인 hex 남발
- **폰트**
  - 크기 계층: title 18/700, body 14/500, label 13/500, caption 12/500, meta 11/500
  - weight 은 500/600/700 외 금지 (400 본문, 900 로고 외)
- **간격 (8의 배수)**
  - 8 / 12 / 16 / 20 / 24 / 28 px 만. 임의 px(7, 13, 22 등) 금지
  - Section padding 20-28px, Card padding 14-20px
- **Border radius**
  - Input 6-8px / Card 10-14px / Section 12-16px / Chip·Badge 16-20px (pill 시 999px)
  - 페이지 헤더·패널 헤더는 `PageShell`/`PanelHeader` 공통 컴포넌트 사용 — 직접 styled 선언 금지
- **Shadow**
  - 가벼운 box-shadow 만 사용 (`0 1px 2px rgba(0,0,0,0.05)` / `0 4px 12px rgba(0,0,0,0.06)`). 임의 커스텀 섀도우 금지

### 8-B 트렌드/고퀄리티 디테일

- **마이크로 인터랙션**: hover transition 0.15s, focus ring = Primary 500 + opacity 0.3 box-shadow, active 시 subtle scale/bg 변화
- **스위치/토글**: 라벨 + 보조 설명 조합, disabled 시 opacity 0.5, `role="switch"` + `aria-checked` 명시
- **빈 상태**: 단순 "No data" 문자열 금지 → Icons.tsx 아이콘 + 1줄 설명 + 명확한 CTA 버튼
- **로딩 상태**: 단순 "Loading..." 금지. spinner 또는 skeleton 사용, 레이아웃 변동(CLS) 없도록 placeholder 높이 고정
- **에러 상태**: 사용자에게 원인 + 해결 힌트. `Failed` 한 마디 금지
- **폼 저장**: AutoSaveField 적용 원칙 (UI_DESIGN_GUIDE.md 섹션 7) — 저장 버튼 없이 debounce 저장, ✓ 뱃지만 2초 페이드. 예외: 청구서 같은 복합 폼
- **긍정 피드백**: **alert, toast.success, window.alert 절대 금지** (헬스체크가 차단). 성공 시 모달 닫기 + 조용한 리프레시

### 8-C 기능 적합성

- **필드 타입 최적화**
  - 날짜 단일·범위: `components/Common/CalendarPicker.tsx` 사용. `<input type="date">` 직접 사용 지양 (허용 시 i18n/tz 일관성 확인)
  - 셀렉트: `PlanQSelect` 강제 (raw `<select>` / react-select 직접 import 금지 — 헬스체크 린트 차단)
  - 전화번호: `PhoneInput`
  - 타임존: `TimezoneSelector`
  - 파일 업로드: `FileUpload`
- **타임존 (PlanQ 필수 규칙)**
  - 모든 사용자 대면 시각은 `hooks/useTimeFormat` 또는 `utils/dateFormat.ts` 경유 — 직접 `toLocaleString()`/`toLocaleDateString()`/`toLocaleTimeString()` 호출 시 반드시 `timeZone` 옵션 포함
  - DB 는 UTC, 표시는 `user.workspace_timezone` 기준
  - 업무 경계(오늘/이번 주/연체) 계산도 워크스페이스 tz (`utils/timezones.ts` `todayInTz` 등 사용)
- **긴 폼**: 4개 이상 섹션이면 탭 분할 (`components/Common/TabComponents.tsx`) 또는 스텝
- **동적 리스트**: 추가/삭제 아이콘 버튼, 빈 배열 CTA, 카운트 배지 (Primary 100 bg + Primary 800 text)
- **체크리스트/설정**: 12개 이상은 카테고리 그룹핑 (Section + subtitle)
- **read-only**: `disabled` input 대신 definition list 스타일 (label + value), 값 없음은 "—" 플레이스홀더

### 8-D 반응형

- `@media (max-width: 768px)` 분기: 3컬럼 → 1컬럼, grid 1fr, padding 감소(24→16), 테이블 → 카드 변환
- 가로 스크롤 금지. 탭 오버플로우는 `overflow-x: auto` + scrollbar 숨김
- 모바일 입력 UX: `inputMode` 속성 (numeric/decimal/email 등), 터치 타겟 44px+ 확보
- 확인 방법: `curl https://dev.planq.kr/<path>` + 빌드 CSS 에 `@media` 분기 grep

### 8-E i18n (필수)

- **하드코딩 텍스트 0건** — 모든 문구 `t('ns:key')` 사용. 검사:
  ```bash
  grep -rEn "(['\"\`])[^'\"\`]*[가-힣][^'\"\`]*\1" dev-frontend/src --include='*.tsx' --include='*.ts' \
    | grep -v -E '//|/\*|\*/|^\s*\*' | grep -v '/locales/'
  ```
  결과 줄 없어야 통과
- ko/en 양쪽 JSON 에 키 존재 확인 (`public/locales/ko`, `public/locales/en`)
- `i18n.ts` 의 `ns` 배열에 신규 네임스페이스 등록 여부

### 8-F 접근성 기본

- `<button>` 에 `type="button"` (submit 아닌 경우)
- 아이콘 전용 버튼은 `title` + `aria-label`
- 토글: `role="switch"` + `aria-checked`
- 색상 대비 AA (텍스트 4.5:1)
- 포커스 ring 제거 금지 (Primary 500 opacity 0.3)

### 8-G 실제 확인 방법 (코드 리뷰만 금지)

- 변경한 페이지를 curl 로 직접 서빙 확인: `curl -s -o /dev/null -w "%{http_code}" https://dev.planq.kr/<path>`
- **구체적 파일:라인 인용**하며 항목별 ✅/❌ 기재 ("LayoutX.tsx:142 에 인라인 hex `#AABBCC` 남음 → COLOR_GUIDE.md 800 으로 교체 필요")
- 부적합 발견 시 즉시 수정 → 재빌드 → 재검증. "다음에 하겠다" 금지
- 빌드 산출물에서 인라인 hex 분포 확인:
  ```bash
  grep -rEn "#[0-9A-Fa-f]{3,8}" dev-frontend/src --include='*.tsx' --include='*.ts' \
    | grep -v -E '//|/\*' | wc -l
  ```
- i18n 하드코딩 검사 (8-E 스니펫)
- 레이아웃 표준 위반 검사 (페이지 루트에 `<Page>`/`<Header>` styled 직접 선언했는지):
  ```bash
  grep -rn "const Page = styled\|const Header = styled" dev-frontend/src/pages
  ```

## 🔥 PlanQ 특수 검증 항목 (사이클 N+38 박제)

PlanQ 는 Vite + React SPA + Node + Python (Q Note) + Sequelize + MySQL + Socket.IO 환경.
일반 e2e plan (Next.js + Supabase) 에 없는 PlanQ 만의 critical 항목 — **검증 시 반드시 추가 확인**.

### A. 멀티테넌트 격리 — `business_id` WHERE 강제 (최우선 보안)

PlanQ 는 Supabase RLS 와 달리 **Sequelize WHERE 절 수동 강제**. 누락 시 cross-tenant 데이터 누출.

- [ ] 신규 mutation 라우트에 `WHERE business_id = ?` 또는 `attachWorkspaceScope` middleware 적용 확인
- [ ] GET 라우트 응답에 다른 워크스페이스 row 안 섞이는지 (raw SQL grep 으로)
- [ ] API test 단계 — 다른 워크스페이스 user 토큰으로 호출 → **403 응답 확인 필수**
- 검증 명령:
  ```bash
  grep -rEn "findAll\(\s*\{\s*where\s*:\s*\{[^}]*\}" dev-backend/routes \
    | grep -v "business_id"  # business_id 없는 쿼리 = 의심
  ```

### B. Socket.IO 실시간 동기화 (CLAUDE.md 운영 안정성 16번 박제)

신규 페이지/라우트 추가 시 4 요소 체크리스트 강제. 누락 시 사용자 호소 "리프레시 안 하면 안 보임" 회귀.

- [ ] **(a) socket 연결 + business room join** — 페이지 mount 시 `s.emit('join:business', businessId)`
- [ ] **(b) backend broadcast** — 데이터 변경 라우트 모두 `io.to('business:${bizId}').emit('<event>', payload)` 호출
- [ ] **(c) frontend listener** — `s.on('<event>', debouncedReload)` 250ms debounce + silentLoad 또는 setState merge
- [ ] **(d) useVisibilityRefresh 안전망** — PWA background → foreground 또는 socket 끊김 시 missed event 회복
- 검증 명령:
  ```bash
  # 신규 라우트 broadcast 누락 검출
  grep -nE "router\.(post|put|delete)" dev-backend/routes/{신규}.js \
    | wc -l  # mutation 수
  grep -nE "io\.to|emit\(" dev-backend/routes/{신규}.js | wc -l  # broadcast 수
  # 두 값 차이 = 누락
  ```

### C. Q Note 별도 FastAPI service (port 8000)

PlanQ backend 는 Node (port 3003) + Python FastAPI (port 8000) 2개 service.

- [ ] Q Note 관련 검증 시 — Node socket.io 와 별개 (window CustomEvent 또는 polling 패턴)
- [ ] PM2 health: planq-dev-backend + **planq-qnote 둘 다 online** 확인
- [ ] q-note FastAPI 자체 헬스체크 — `curl -sf http://localhost:8000/health`
- [ ] qnote_sessions 변경 시 → window.dispatchEvent('qnote-session-created/updated') 패턴 박제 (사이클 N+35)

### D. Visibility 4단계 (L1~L4) + 권한 4-Layer

PlanQ 는 Supabase RLS 와 달리 visibility + 권한 매트릭스 매우 복잡.

**Visibility 4단계 (VISIBILITY_VOCABULARY.md):**
- L1 개인 / L2 팀 비공개 / L3 워크스페이스 / L4 외부
- [ ] 신규 자산 (file/post/kb/session) 추가 시 visibility 컬럼 + filter 강제
- [ ] visibility 변경 (L1 → L2/L3/L4 promote) 시 audit log + 권한 재검증

**권한 4-Layer (PERMISSION_MATRIX.md):**
1. **Role** — owner / admin / member / client
2. **워크스페이스 토글** — financial / schedule / client_info × all / pm
3. **멤버별 메뉴 권한** — 11 메뉴 × 3 레벨 (none/read/write) 기본 write (열린 문화)
4. **자원 owner** — Invoice.owner_user_id 등

- [ ] 신규 라우트 추가 시 4-Layer 모두 검증
- [ ] `requireMenu(menu, level)` middleware 적용 확인 (qbill / weekly_team 등)
- [ ] client role 차단 라우트 — `routes/access_scope.js` 의 `attachWorkspaceScope` + `isMemberOrAbove`

### E. State hydration (Vite SPA — Next.js SSR 과 다름)

PlanQ 는 SSR 없음 → server-side hydration mismatch 이슈는 X.
대신 **localStorage / sessionStorage / cookie state 와 첫 render 불일치** 점검:

- [ ] theme / language / scroll 위치 / sidebar 토글 복원 — useEffect 안에서 `localStorage.getItem` 호출 + Skeleton 또는 default state 로 첫 paint
- [ ] socket 연결 직후 vs 직전 state 불일치 — `useVisibilityRefresh` 가 server fresh 덮어쓰기 (memory `feedback_visibility_refresh_server_fresh.md`)
- [ ] AuthContext 초기화 — `getAccessToken()` 시점 race 점검

## 10단계 — Playwright MCP e2e 브라우저 검증 (사이클 N+39 박제)

API 단위 검증 (3·5단계) 으로 백엔드는 정상이라 보고했는데, 실제 브라우저에선 클릭이 안 먹거나 모달이 안 뜨는 회귀를 차단하기 위함. Playwright MCP 를 사용해 진짜 브라우저 흐름까지 자동 검증.

### 언제 실행

- `/검증 --e2e` 명시 시 (항상)
- 큰 사이클 마지막 검증 (대규모 기능 추가, 라우팅 변경, 인증 변경, 결제·서명·OTP 흐름 등)
- 소규모 작업은 1~9단계만 OK. 10단계 강제 X

### 사용 도구

**Playwright MCP** (`mcp__playwright__*`) — Anthropic 공식 MCP 서버, **무료**. 별도 결제·구독 없음.

핵심 도구:
- `browser_navigate` — URL 이동
- `browser_snapshot` — 현재 페이지 accessibility tree 스냅샷
- `browser_click` / `browser_type` / `browser_fill_form` — 입력·클릭
- `browser_evaluate` — `(page) => ...` 자유 JS 평가
- `browser_console_messages` — 콘솔 에러 수집
- `browser_network_requests` — 네트워크 요청 수집

### 검증 흐름

1. **dev 서버 접속** — `browser_navigate("https://dev.planq.kr/login")`
2. **로그인** — owner / member / client 역할별 (e2e 전용 test account 별도 운용. 검증 끝나면 데이터 원복)
3. **이번 사이클 변경된 화면 진입** — 변경 페이지 경로 (라우트 변경했으면 옛 경로 404 + 새 경로 200 둘 다)
4. **핵심 인터랙션 1~3개**:
   - 입력 → 저장 → ✓ 자동저장 뱃지 확인
   - 모달 열기 → 입력 → 확인 → 닫힘 확인
   - 리스트 row 클릭 → drawer 열림 → URL `?task=N` 싱크 확인
5. **콘솔 에러 수집** — `browser_console_messages` 결과에 `error` level 0건 확인 (warn 은 허용)
6. **네트워크 에러** — `browser_network_requests` 의 status ≥400 응답 검토 (의도된 401/403 외 모두 0)
7. **검증 끝 — test 계정 데이터 원복** (CLAUDE.md `feedback_test_data_restore.md` 박제)

### LLM auto-fix 정책 (사용자 박제, 절대 위반 금지)

테스트 실패 시 LLM 이 **자동으로 고쳐도 되는 영역 = selector / waitFor / timeout 한정**. 비즈니스 로직 자동 수정 절대 금지.

| 실패 유형 | LLM auto-fix 허용 | 행동 |
|----------|-------------------|------|
| selector 못 찾음 (DOM 변경) | ✅ | accessibility tree 다시 읽어 selector 갱신 |
| `waitFor` timeout (느린 응답) | ✅ | timeout 늘림 또는 다른 visible 상태 대기 |
| 텍스트 변경 (i18n 키 갱신) | ✅ | i18n JSON 에서 정확한 텍스트 확인 후 적용 |
| API 응답 변경 | ❌ | **사용자에게 보고. 자동 수정 금지** |
| 비즈니스 로직 차이 (계산식·권한·status 전이) | ❌ | **사용자에게 보고. 자동 수정 금지** |
| 결제·서명·OTP 흐름 | ❌ | **사용자에게 보고. test data 도 함부로 만지지 않음** |
| visibility / 권한 자동 우회 | ❌ | **즉시 중단. 보안 위반** |

LLM 이 selector/waitFor 만 고치는 이유 — 화면 DOM·텍스트는 사이클마다 자연스레 바뀜. 매번 사람이 selector 일일이 갱신할 시간 없음. 하지만 **테스트가 실패한 게 진짜 비즈니스 회귀일 때 LLM 이 "테스트 기대값을 바꿔서 통과시키는" 안티패턴은 절대 금지**. PR/사이클 단위 회귀 보호망 자체가 무너짐.

### 무료 한도 안에서 안정적 운용

- Playwright MCP 자체는 무료. dev 서버 로컬 호출이라 외부 비용 없음
- e2e test 계정 (owner/member/client) — DB seed 로 박제, 검증 후 원복 (`try/finally` 패턴)
- screenshot 저장은 dev 서버 `/tmp/e2e-snapshots/` 휘발성. 운영 자료 X
- 운영 도메인 (`planq.kr`) 으로는 절대 e2e 실행 X — dev 만
