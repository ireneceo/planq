# 검증 명령어

구현 완료 후 반드시 실행하는 검증 프로세스.
모든 단계를 실제로 실행해야 하며 확인했습니다만 말하는 것은 금지.
각 파트별 전문가 관점에서 체크하고 실제 결과를 보여줘야 한다.

## 검증 체계: 5개 파트 전문가 관점

개발기획: 요구사항 충족, 데이터 흐름 정합성, 엣지케이스 (3단계, 6단계)
백엔드 개발: API 정상, DB 정합성, 보안, 에러 처리 (2단계, 3단계)
프론트엔드 개발: 빌드, 타입 에러, 렌더링, 상태 관리 (1단계, 4단계)
UI/UX 기획 디자인: 디자인 통일성, 인터랙션, 빈 상태, 로딩 (8단계)
QA: 유저 흐름, 역할별, 반응형, 연관 영향 (5단계, 7단계, 9단계)

## 9단계 검증 (건너뛰기 절대 금지)

1단계 빌드: cd /opt/planq/dev-frontend && npm run build (run_in_background), 타입에러 0건 확인
2단계 백엔드: pm2 restart planq-dev-backend, 에러로그 확인, 인증/응답형식 체크
3단계 API 실호출: Node 스크립트로 실제 HTTP 호출 (포트 3003). Write->Read 왕복, 에러케이스, 권한별 접근
4단계 프론트 렌더링: curl로 HTML/JS 번들 확인 (https://dev.planq.kr)
5단계 유저 흐름: 역할별 로그인->기능->저장->재조회 순서대로 API 호출
6단계 요구사항 대조: 요청 항목별 구현 완료 여부 표시
7단계 연관 영향: 공통 컴포넌트 사용처 확인, 부작용 없음 확인
8단계 UI/UX: 색상/폰트/간격 통일, 로딩/빈상태/에러 인터랙션
9단계 크로스페이지: 네비게이션, SPA 라우팅, 역할별 리다이렉트

## API 테스트 패턴

```bash
cd /opt/planq/dev-backend
node test-xxx.js    # Login → API 호출 → 검증
rm test-xxx.js      # 반드시 삭제
```

## 절대 금지
코드 확인만으로 정상이라 하지 말고 실제 호출 결과를 보여라.
API 호출 없이 검증 완료 금지. 빌드 안 돌리고 빌드 성공 금지.
에러 있는데 완료 금지. UI/UX 건너뛰기 금지.
